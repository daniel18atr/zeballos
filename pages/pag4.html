Ciclo de Vida del Desarrollo de Software (SDLC)

Qué es el SDLC y por qué importa

El SDLC es el conjunto de fases, procesos y prácticas que guían la creación, entrega y mantenimiento de software. Su objetivo es entregar software útil y de calidad, a tiempo y dentro del presupuesto, minimizando riesgos y facilitando la comunicación entre stakeholders.

Beneficios claros: menor riesgo, mayor calidad, trazabilidad, previsibilidad en costos/tiempos y mayor facilidad para mantener y escalar el sistema.

Fases principales (descripción detallada)

  1. Planificación y análisis de viabilidad

    Actividades:

    • Definir alcance inicial, objetivos de negocio y stakeholders.
    • Evaluar viabilidad técnica, económica y de calendario.
    • Identificar restricciones, dependencias y riesgos iniciales.

    Entregables:

    • Documento de alcance / Business Case.
    • Análisis de viabilidad y plan de proyecto inicial.

    Importancia:

    • Evita arrancar sin objetivos claros; permite estimaciones realistas y priorizar lo esencial.
  2. Recolección y análisis de requisitos

    Actividades:

    • Identificar y conversar con stakeholders (usuarios, negocio, ops).
    • Capturar requisitos funcionales (qué debe hacer) y no funcionales (seguridad, rendimiento, disponibilidad).
    • Priorizar requisitos y definir criterios de aceptación.

    Entregables:

    • Especificación de Requisitos (SRS) o backlog de producto (en ágil).
    • Historias de usuario y criterios de aceptación.
    • Matriz de trazabilidad requisitos casos de prueba.

    Importancia:

    • Requisitos mal hechos = re-trabajo costoso. Buena especificación reduce malentendidos y asegura que el equipo construya lo correcto.
  3. Diseño (arquitectura y diseño detallado)

    4. Actividades:

    • Diseño de arquitectura: componentes, límites, patrones (microservicios, monolito, etc.).
    • Diseño de datos: modelo ER, esquema de DB.
    • Diseño de interfaces: wireframes, API contracts (OpenAPI/Swagger).
    • Diseño detallado: secuencias, clases, contratos.

    Entregables:

    • Documento de arquitectura (C4, diagrams UML).
    • Especificaciones de API y wireframes/prototipos.
    • Decisiones de tecnología (stack), criterios de escalado.

    Importancia:

    • Reduce dependencias ocultas y facilita la escalabilidad y mantenibilidad. Un mal diseño provoca “technical debt”.
  4. Implementación / Codificación

    5. Actividades:

    • Programación según diseño.
    • Revisión de código (code review / pull requests).
    • Gestión de versiones (git branching: feature/PR → main).
    • Integración continua (CI): builds y tests automatizados.

    Entregables:

    • Repositorio con código, tests y documentación técnica mínima.
    • Artefactos de build (paquetes, contenedores).

    Importancia:

    • Fase donde se materializa el producto. Buenas prácticas (estándares, revisiones, CI) reducen errores y facilitan entregas rápidas.
  5. Pruebas (QA: verificación y validación)

    6. Tipos y actividades:

    • Unit tests, integration tests, system tests, smoke tests.
    • Pruebas de aceptación (UAT) con usuarios.
    • Pruebas no funcionales: performance, seguridad, usabilidad, accesibilidad.
    • Automatización vs pruebas manuales; pipelines de test automatizados.

    Entregables:

    • Plan de pruebas, casos y resultados.
    • Reportes de defectos y métricas (defect density, pass rate).

    Importancia:

    • Detectar fallos antes del despliegue reduce costes. Pruebas continuas aumentan la confianza en el release.
  6. Despliegue (release)

    7. Actividades:

    • Preparar release: empaquetado, migraciones de DB, configuración.
    • Estrategias de despliegue: blue-green, canary, rolling, feature flags.
    • Automatización con CD (Continuous Delivery/Deployment).
    • Preparación de runbooks y rollback.

    Entregables:

    • Release notes, pipelines CD, scripts de deploy.
    • Plan de rollback y validación post-deploy.

    Importancia:

    • Despliegues seguros y repetibles reducen downtime y riesgo de incidentes en producción.
  7. Mantenimiento y operación

    8. Actividades:

    • Soporte, corrección de bugs, parches de seguridad.
    • Evolutivos: nuevas funcionalidades y mejoras.
    • Monitoreo (logs, métricas, alertas) y gestión de incidentes (MTTR).
    • Gestión de deuda técnica y actualizaciones.

    Entregables:

    • Tickets/incidencias resueltas, actualizaciones de documentación, SLA reports.

    Importancia:

    • La mayor parte del coste de software está en mantenimiento; operar bien asegura continuidad del servicio.

Modelos de SDLC (comparativa rápida)

  • Waterfall: fases secuenciales. Bueno si requisitos estables. Riesgo: poco flexible.
  • Iterativo / Incremental: ciclos que entregan partes funcionales. Menos riesgo, entrega valor temprana.
  • Spiral: enfoque por riesgos; bueno para proyectos grandes y de alto riesgo.
  • V-Model: enfatiza pruebas en paralelo con diseño; buena trazabilidad.
  • Ágil (Scrum, Kanban): entregas frecuentes, backlog y sprints; alta adaptabilidad.
  • DevOps / Continuous: fusiona desarrollo y operaciones; integración y despliegue continuos, automatización y feedback constante.

Cuándo usar cada uno: si el entorno cambia rápido y se necesita validar con usuarios, ágil/DevOps. Si requisitos regulados y fijos, modelos más documentados (V/Waterfall) pueden servir.

Roles clave en SDLC

  • Product Owner / Sponsor
  • Business Analyst
  • Arquitecto de Software
  • Desarrolladores
  • QA / Testers
  • DevOps / SRE
  • Seguridad (SecOps)
  • Usuarios finales / Stakeholders

Entregables típicos por fase (resumen)

  • Planificación: Business case, cronograma.
  • Requisitos: SRS, backlog.
  • Diseño: diagramas, API specs.
  • Implementación: código, builds.
  • Pruebas: reportes, casos.
  • Despliegue: scripts, release notes.
  • Mantenimiento: parches, incident reports.

Métricas útiles (KPIs)

  • Lead time (commit → deploy).
  • Cycle time por feature.
  • Coverage de tests unitarios.
  • Defect density / escape rate (defectos en producción).
  • MTTR (Mean Time To Repair).
  • Uptime / SLA compliance.
  • Tasa de fallos en despliegue.

Buenas prácticas (prácticas recomendadas)

  • Shift-left: integrar pruebas y seguridad desde requisitos/diseño.
  • Automatizar CI/CD y pruebas repetibles.
  • Versionar infra como código (IaC).
  • Revisiones de diseño y código regulares.
  • Mantener trazabilidad de requisitos a casos de prueba.
  • Usar feature flags para controlar lanzamientos.
  • Documentación ligera pero suficiente (APIs, runbooks).
  • Monitoreo y alertas proactivas en producción.

Errores comunes a evitar

  • Empezar a codificar sin requisitos claros.
  • Falta de pruebas automatizadas.
  • No planear despliegues (no tener rollback).
  • Ignorar seguridad y rendimiento hasta producción.
  • No medir (sin métricas no se mejora).

Checklist rápido para empezar un proyecto bien

  1. Definir objetivo de negocio y criterios de éxito.
  2. Identificar stakeholders y priorizar requisitos.
  3. Elegir modelo SDLC y stack tecnológico.
  4. Definir pipeline CI/CD mínimo viable.
  5. Crear plan de pruebas y criterios de aceptación.
  6. Implementar monitoreo y runbooks antes del primer release.
